## LexFlow · 多智能体法律决策引擎

> 基于 LangGraph 的生产级法律咨询多智能体系统

**LawBot+** 是一个以法律咨询为核心场景的 AI Agent 应用，采用 **多智能体协作 + 混合 RAG 检索 + 人机协同审核 (HITL)** 的技术架构。系统由 FastAPI 后端（LangGraph 工作流引擎 + MCP 工具协议）、Next.js 前端以及 Docker 中间件组成，可一键部署至生产环境。

## 1. 项目概述

### 1.1 项目简介

LawBot+ 是一个基于大语言模型（LLM）的智能法律咨询系统，通过多Agent协作和RAG（检索增强生成）技术，为用户提供专业的法律咨询服务。

### 1.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| **后端框架** | FastAPI + LangGraph |
| **LLM** | 阿里云百炼 API (Qwen 系列) |
| **向量模型** | BGE (BAAI/bge-small-zh-v1.5) |
| **重排模型** | BGE Reranker (BAAI/bge-reranker-base) |
| **数据库** | PostgreSQL + Redis |
| **异步任务** | Celery |
| **前端** | Next.js 15 + React + TailwindCSS |
| **AI SDK** | LangChain（后端 LLM 调用） |

## 2. Agent 架构

### 2.1 Agent 系统概述

LawBot+ 采用 **LangGraph** 实现多Agent协作状态机，每个Agent负责特定任务，通过状态共享实现协作。

### 2.2 Agent 类型与职责

| Agent | 文件 | 职责 | 输入 | 输出 |
|-------|------|------|------|------|
| **Planner** | `planner.py` | 任务规划、意图识别、敏感过滤 | 用户问题 | 改写后的检索式 |
| **Researcher** | `researcher.py` | 法律知识检索、结果评估 | 检索式 | 检索结果列表 |
| **Analyst** | `analyst.py` | 法律逻辑推理、结论生成。检索为空时直接返回"知识库无相关记录"，禁止复用旧答案 | 检索结果 | 法律分析结果 |
| **Reviewer** | `reviewer.py` | 基于检索指标计算置信度与风险评估，决定是否触发HITL | 分析结果 | 置信度、风险等级、是否需要审核 |
| **Tool Agent** | `tool_agent.py` | 工具决策与调用 | 当前状态 | 工具执行结果 |
| **Memory** | `memory_manager.py` | 短期会话记忆管理（含**跨会话状态隔离**），更新与检索；长期记忆为 TODO | 当前状态 | 更新后的状态 |

### 2.3 Agent 状态定义

核心状态模型 `AgentState`（`src/agents/state.py`）：

```python
class AgentState(BaseModel):
    # ===== 对话上下文 =====
    session_id: str = Field(description="会话ID")
    user_input: str = Field(description="用户原始输入")
    rewritten_query: Optional[str] = Field(default=None, description="改写后的检索query")
    tokenized_query: Optional[list[str]] = Field(default=None, description="分词后的检索词列表")

    # ===== 消息历史 =====
    messages: list[dict] = Field(default_factory=list, description="对话历史")

    # ===== 当前执行状态 =====
    current_agent: Literal["planner", "researcher", "analyst", "reviewer", "memory", "tools"] = Field(
        default="planner", description="当前活跃的Agent"
    )

    # ===== 检索结果 =====
    retrieved_docs: list[dict] = Field(default_factory=list, description="检索到的文档")
    reranked_docs: list[dict] = Field(default_factory=list, description="重排后的文档")
    relevance_check_passed: bool = Field(default=True, description="相关性检查是否通过")

    # ===== 分析与推理 =====
    analysis_result: Optional[str] = Field(default=None, description="分析师推理结果")
    reasoning_chain: list[str] = Field(default_factory=list, description="推理链")

    # ===== 审核与HITL =====
    needs_review: bool = Field(default=False, description="是否需要人工审核")
    confidence_score: float = Field(default=1.0, ge=0.0, le=1.0, description="置信度")
    risk_level: Literal["low", "medium", "high", "critical"] = Field(
        default="low", description="风险等级"
    )

    # ===== 最终输出 =====
    final_answer: Optional[str] = Field(default=None, description="最终回答")
    sources: list[dict] = Field(default_factory=list, description="引用来源")

    # ===== 工具调用 =====
    tool_calls: list[dict] = Field(default_factory=list, description="工具调用记录")
    tool_results: list[dict] = Field(default_factory=list, description="工具执行结果")
    needs_tool: bool = Field(default=False, description="是否需要调用工具")

    # ===== 记忆上下文 =====
    memory_context: dict = Field(default_factory=dict, description="长期记忆上下文")

    # ===== 元数据 =====
    task_id: Optional[str] = Field(default=None, description="任务ID")
    created_at: datetime = Field(default_factory=datetime.now)
    updated_at: datetime = Field(default_factory=datetime.now)

    # ===== 错误处理 =====
    error: Optional[str] = Field(default=None, description="错误信息")
    retry_count: int = Field(default=0, description="重试次数")
```

> **说明**：
> - `risk_level` 类型定义为 `Literal["low", "medium", "high", "critical"]`，`critical` 属于预留等级。
> - `confidence_score` 描述为**置信度**（0.0-1.0），代表整体回答质量，由检索指标（rerank_score）计算得出，与知识匹配度含义相近但基于不同计算方式。
> - 字段分组顺序与代码中 `AgentState` 的实际定义顺序一致。
> - `tokenized_query` 字段保存分词后的检索词列表（来自 BM25 分词结果），代码中已实际存在。
> - `task_id` 字段位于元数据分组最后，与代码中的实际顺序一致。

## 3. RAG 架构

### 3.1 RAG 系统概述

LawBot+ 采用 **混合检索** 策略，结合向量检索、BM25关键词检索和交叉编码器重排，提供高质量的法律知识检索。

### 3.2 RAG 组件

| 组件 | 文件 | 功能 | 模型 |
|------|------|------|------|
| **知识库接口** | `knowledge_base.py` | RAG统一入口、查询改写 | - |
| **混合检索管道** | `hybrid_search.py` | 检索编排、分数融合 | - |
| **向量嵌入** | `embedding.py` | 文本向量化 | BAAI/bge-small-zh-v1.5 (512维) |
| **BM25检索** | `bm25_search.py` | 关键词全文检索（jieba分词 + 法律词典） | rank_bm25.BM25Okapi + jieba |
| **重排模型** | `reranker.py` | 交叉编码器重排 | BAAI/bge-reranker-base |
| **查询改写器** | `query_rewriter.py` | 自然语言→法言法语 | LLM (Qwen) |

```python
@router.post("/upload", response_model=UploadResponse)
async def upload_knowledge_file(file: UploadFile = File(...)):
    """上传知识库文件

    支持格式：
    - 单个文档：{"title": "...", "content": "...", "article": "第X条", "source": "法律名称"}
    - 文档列表：[{...}, {...}]
    - 标准格式：{"data": [{...}]}
    """
    # 1. 检查文件类型
    if not file.filename.endswith('.json'):
        raise HTTPException(status_code=400, detail="只支持 JSON 格式文件")

    # 2. 读取文件内容
    content = await file.read()
    file_content = content.decode('utf-8')

    # 3. 调用上传服务处理
    result = await knowledge_base_uploader.process_upload(file_content, file.filename)

    return UploadResponse(
        success=True,
        message=result.message,
        total_records=result.total_records,
        new_records=result.new_records,
        duplicates=result.duplicates,
        merged_records=result.merged_records
    )

@router.post("/batch", response_model=UploadResponse)
async def batch_upload_knowledge(request: BatchUploadRequest):
    """批量上传知识文档（JSON请求体方式）"""
    file_content = json.dumps(request.documents, ensure_ascii=False)
    filename = f"batch_{request.source_name or 'documents'}.json"
    result = await knowledge_base_uploader.process_upload(file_content, filename)
    return UploadResponse(success=True, message=result.message, ...)
```

#### 处理流程（核心方法）

```python
async def process_upload(self, file_content: str, filename: str) -> UploadResult:
    """处理上传的文件"""

    # ===== Step 1: 解析文件内容 =====
    # 支持多种 JSON 格式：
    # - 数组格式：[ {...}, {...} ]
    # - data 字段：{ "data": [...] }
    # - documents 字段：{ "documents": [...] }
    # - records 字段：{ "records": [...] }
    # - 单文档：{ ... }
    data = self._parse_file_content(file_content, filename)

    # ===== Step 2: 转换为标准格式 =====
    # 标准化字段映射：
    # title ← name, 法条名称
    # content ← text, 条款内容
    # source ← 法律名称, law
    # article ← 条款号, clause
    documents = self._normalize_documents(data)

    # ===== Step 3: 保存上传文件到 json/ 目录 =====
    # 文件命名规则：uploaded_{safe_name}_{timestamp}.json
    self._save_to_json_dir(documents, filename)

    # ===== Step 4: 运行完整去重流程 =====
    # 从 json/ 目录加载所有文件，进行两层去重：
    # - Layer1: 精确去重（source|article 唯一键 + MD5 哈希）
    # - Layer2: 智能合并（同条款多来源，按优先级保留最权威版本）
    self._run_full_dedup()

    # ===== Step 5: 重新索引向量数据库 =====
    # 加载最新的 deduplicated_*.json，构建：
    # - BM25 索引（关键词倒排索引）
    # - 向量索引（BGE 模型 512 维向量）
    latest_file = self._get_latest_processed_file()
    if latest_file:
        await self._reindex_knowledge_base(latest_file)

    return UploadResult(
        success=True,
        total_records=stats["total"],
        new_records=new_docs_count,
        message=f"成功添加 {new_docs_count} 条法律条文，知识库已更新"
    )
```

### 5.5 BM25检索 (bm25_search.py)

**文件路径**: `src/rag/bm25_search.py`

**核心类**: `BM25Search`

**特点**:
- 基于 rank_bm25 库
- 基于 jieba 精确模式分词，针对中文法律文本优化
- 加载法律领域自定义词典（无过错责任、抵押权、追及效力、连带责任、善意取得、举证责任、加班费、劳动合同、违约责任、诉讼时效共 10 个专有名词）
- 过滤标点、纯数字及长度 < 2 的词条，提升检索质量
- `k1`、`b` 参数从 `settings` 读取，支持运行时配置
- 支持过滤搜索

```python
def _build_legal_dictionary() -> List[str]:
    """构建法律领域自定义词典，供初始化时加载到 jieba。"""
    return [
        "无过错责任", "抵押权", "追及效力", "连带责任", "善意取得",
        "举证责任", "加班费", "劳动合同", "违约责任", "诉讼时效",
    ]


class BM25Search:
    def __init__(self, k1: float | None = None, b: float | None = None) -> None:
        settings = get_settings()
        self.k1 = k1 if k1 is not None else settings.bm25_k1
        self.b = b if b is not None else settings.bm25_b
        self.bm25: BM25Okapi | None = None
        for word in _build_legal_dictionary():
            jieba.add_word(word)

    def _tokenize(self, text: str) -> List[str]:
        tokens = jieba.lcut(text, cut_all=False)
        return [t for t in tokens if t.strip() and not t.strip().isdigit() and len(t) >= 2]

    def search(self, query: str, top_k: int = 10) -> List[Tuple[int, float]]:
        tokenized_query = self._tokenize(query)
        scores = self.bm25.get_scores(tokenized_query)
        doc_scores = list(enumerate(scores))
        doc_scores.sort(key=lambda x: x[1], reverse=True)
        return doc_scores[:top_k]
```

```python
    ) -> List[Dict]:
        # 1. 查询改写
        if use_rewrite:
            rewritten = await self.query_rewriter.rewrite(query)
            search_query = rewritten.get("rewritten_query", query)

        # 2. 执行混合检索
        results = await self.search_pipeline.search(
            query=search_query,
            top_k=top_k,
            filter_func=filter_func
        )
        return results
```

```python
async def update_short_term(self, state: AgentState) -> AgentState:
    session_key = f"session:{state.session_id}"
    existing_history = self.short_term_memory.get(session_key, [])

    # 新会话的第一条消息时，清空历史防止跨 session 污染
    if not existing_history:
        self.short_term_memory[session_key] = []

    history = self.short_term_memory.get(session_key, [])
    history.append({
        "user_input": state.user_input,
        "analysis_result": state.analysis_result,
        "final_answer": state.final_answer,
        "timestamp": str(state.created_at)
    })

    # 最多保留最近 10 条记录
    if len(history) > self.max_history:
        history = history[-self.max_history:]

    self.short_term_memory[session_key] = history

    # 构建上下文摘要（仅包含当前会话）
    if history:
        state.memory_context = {
            "summary": f"用户在此会话中问了{len(history)}个问题",
            "recent_topics": [h.get("user_input", "")[:50] for h in history[-3:]]
        }

    return state
```

### 8.3 核心代码 (deduplication.py)

**文件路径**: `src/rag/deduplication.py`

**核心类**: `LegalDeduplicator`

```python
@dataclass
class LegalDocument:
    """法律文档数据结构"""
    title: str
    content: str
    source: str
    article: str
    domain: str = "民事"
    doc_type: str = "law"
    keywords: List[str] = field(default_factory=list)

    @property
    def unique_key(self) -> str:
        """唯一标识键: source|article"""
        return f"{self.source}|{self.article}"

    def compute_hash(self) -> str:
        """计算内容哈希用于精确去重"""
        content_str = f"{self.source}|{self.article}|{self.content}"
        return hashlib.md5(content_str.encode()).hexdigest()


class LegalDeduplicator:
    """企业级法律条文去重器"""

    SOURCE_PRIORITY = {
        "law": 1,           # 法律原文优先
        "interpretation": 2, # 司法解释次之
        "regulation": 3,     # 行政法规第三
        "default": 4
    }

    def layer1_exact_deduplication(self) -> int:
        """第一层：精确去重"""
        seen_keys: Set[str] = set()
        seen_hashes: Set[str] = set()
        unique_docs: List[LegalDocument] = []
        duplicates = 0

        for doc in self.documents:
            key = doc.unique_key
            doc_hash = doc.compute_hash()

            if key in seen_keys or doc_hash in seen_hashes:
                duplicates += 1
                continue

            seen_keys.add(key)
            seen_hashes.add(doc_hash)
            unique_docs.append(doc)

        self.documents = unique_docs
        return duplicates

    def layer2_smart_merge(self) -> int:
        """第二层：智能合并"""
        article_groups = defaultdict(list)
        for doc in self.documents:
            article_num = self._extract_article_number(doc.article)
            if article_num:
                key = f"{doc.source}|{article_num}"
                article_groups[key].append(doc)

        merged_docs = []
        merged_count = 0

        for key, docs in article_groups.items():
            if len(docs) == 1:
                merged_docs.append(docs[0])
            else:
                # 多条相同条款，智能合并
                merged = self._merge_documents(docs)
                merged_docs.append(merged)
                merged_count += 1

        self.documents = merged_docs
        return merged_count

    def _merge_documents(self, docs: List[LegalDocument]) -> LegalDocument:
        """合并多条相同条款 - 按来源优先级+内容长度排序"""
        docs_sorted = sorted(
            docs,
            key=lambda x: (
                self.SOURCE_PRIORITY.get(x.doc_type, 99),
                -len(x.content)  # 内容越长越完整
            )
        )
        best = docs_sorted[0]
        # 合并关键词
        all_keywords = set()
        for doc in docs:
            all_keywords.update(doc.keywords)
        best.keywords = list(all_keywords)[:10]
        return best

    def layer3_quality_check(self) -> DeduplicationReport:
        """第三层：质量检查（代码中独立实现，文档原版未描述）"""
        report = self.report

        # 1. 条款编号格式校验（不匹配 "第X条" 等格式的记录）
        invalid_articles = []
        for doc in self.documents:
            if not re.match(r'第[\d一二三四五六七八九十百]+[条款项章节节]', doc.article):
                invalid_articles.append((doc.article, doc.source))
        if invalid_articles:
            report.quality_issues.append(f"Found {len(invalid_articles)} articles with invalid format")

        # 2. 各法律覆盖率分析
        coverage = self._analyze_coverage()
        report.coverage_rate = coverage

        # 3. 缺失条款检测（检测条款号间隔 > 10 的区间）
        missing = self._detect_missing_articles()
        report.missing_articles = missing

        # 4. 短内容过滤（< 20 字符的条文）
        short_content = [d for d in self.documents if len(d.content) < 20]
        if short_content:
            report.quality_issues.append(f"Found {len(short_content)} articles with short content")

        return report

    def _analyze_coverage(self) -> Dict[str, float]:
        """分析各法律覆盖率（基于预期条款数计算百分比）"""
        expected_counts = {
            "中华人民共和国民法典": 1260,
            "中华人民共和国劳动法": 107,
            "中华人民共和国劳动合同法": 89,
            "中华人民共和国公司法": 218,
            "中华人民共和国消费者权益保护法": 69,
            "中华人民共和国道路交通安全法": 115,
            "中华人民共和国刑法": 452,
            "中华人民共和国刑事诉讼法": 127,
            "中华人民共和国民事诉讼法": 112,
        }
        law_counts: Dict[str, int] = defaultdict(int)
        for doc in self.documents:
            law_counts[doc.source] += 1
        coverage = {}
        for law, expected in expected_counts.items():
            actual = law_counts.get(law, 0)
            coverage[law] = min(100.0, (actual / expected) * 100) if expected > 0 else 0.0
        return coverage

    def _detect_missing_articles(self) -> Dict[str, List[str]]:
        """检测条款号间隔大于10的缺失区间"""
        missing = {}
        law_articles: Dict[str, Set[str]] = defaultdict(set)
        for doc in self.documents:
            law_articles[doc.source].add(doc.article)
        minguo_articles = law_articles.get("中华人民共和国民法典", set())
        if minguo_articles:
            article_nums = []
            for art in minguo_articles:
                match = re.search(r'第(\d+)条', art)
                if match:
                    article_nums.append(int(match.group(1)))
            article_nums.sort()
            gaps = []
            for i in range(len(article_nums) - 1):
                diff = article_nums[i + 1] - article_nums[i]
                if diff > 10:  # 超过10条的间隔可能是缺失
                    gaps.append(f"{article_nums[i]}-{article_nums[i+1]}")
            if gaps:
                missing["中华人民共和国民法典"] = gaps
        return missing
```

## 9. 会话历史存储

### 9.1 设计概述

LawBot+ 采用**三层降级存储**的方案持久化会话历史（`src/api/session_store.py`），按优先级依次为：Redis → PostgreSQL → InMemoryStore（内存兜底）。

| 存储层 | 技术选型 | 职责 | TTL |
|--------|----------|------|-----|
| **第一层（优先）** | Redis | 活跃会话消息、会话列表索引 | 30天 |
| **第二层（降级）** | PostgreSQL | Redis 不可用时持久化会话数据 | 永久 |
| **第三层（最终兜底）** | 内存（InMemoryStore） | 所有存储层均不可用时兜底 | 进程生命周期 |

**数据流向**：

```
用户发送消息
     │
     ▼
前端 page.tsx → POST /chat
     │
     ▼
后端 main.py:chat()
     │  1. 执行 LangGraph 工作流
     │  2. 获取结果
     │  3. 查询已有会话（session_store.get_conversation）
     │  4. 追加新消息（existing_messages + [user, assistant]）
     │  5. 保存完整历史（session_store.save_conversation）
     │
     ▼
三层存储（Redis优先 → PostgreSQL降级 → 内存兜底）
     │
     ▼
前端刷新 → GET /sessions → 显示历史会话列表
```

### 9.2 核心问题修复

**问题**：原实现中每次只保存最后一对消息，刷新页面后历史全部丢失。

**修复方案**：
1. 后端 `POST /chat` 执行完成后，查询现有会话，**追加**新消息，再保存到三层存储（Redis → PostgreSQL → 内存）
2. 前端仅负责调用 `/api/chat` 接口，**会话保存完全由后端处理**，前端不再单独调用 PUT 接口

### 9.5 SessionStore 实现

**文件路径**: `src/api/session_store.py`

```python
class SessionStore:
    """三层会话存储（Redis优先 → PostgreSQL降级 → 内存兜底）"""
    KEY_PREFIX = "lawbot:session:"
    LIST_KEY = "lawbot:sessions:list"

    def __init__(self):
        self.redis = None
        self._use_redis = None  # None=未检测, True=使用Redis, False=不使用Redis
        self.pg_store = PostgresSessionStore()
        self.memory = InMemoryStore()

    async def connect(self):
        """检测并连接 Redis"""
        if self._use_redis is not None:
            return
        try:
            import redis.asyncio as redis_async
            from src.config import get_settings
            settings = get_settings()
            self.redis = redis_async.from_url(settings.redis_url, ...)
            await self.redis.ping()
            self._use_redis = True
        except Exception as e:
            self._use_redis = False
            logger.warning(f"Redis不可用，将降级到PostgreSQL: {e}")

    def _auto_title(self, messages: List[dict]) -> str:
        """自动生成会话标题（取首条用户消息前30字）"""
        for msg in messages:
            if msg.get("role") == "user":
                content = msg["content"]
                return content[:30] + ("..." if len(content) > 30 else "")
        return "新会话"

    def _build_data(self, session_id: str, messages: List[dict], title: str = None) -> dict:
        """构建存储数据"""
        title = title or self._auto_title(messages)
        return {
            "session_id": session_id,
            "title": title,
            "messages": messages,
            "updated_at": datetime.now().isoformat(),
            "created_at": datetime.now().isoformat(),
        }

    async def save_conversation(self, session_id: str, messages: List[dict], title: str = None):
        """保存会话"""
        await self.connect()
        data = self._build_data(session_id, messages, title)

        # 第一层：Redis（30天TTL，ZSET维护会话列表）
        if self._use_redis:
            try:
                key = f"{self.KEY_PREFIX}{session_id}"
                await self.redis.set(key, json.dumps(data, ensure_ascii=False), ex=86400 * 30)
                await self.redis.zadd(self.LIST_KEY, {session_id: datetime.now().timestamp()})
                return
            except Exception as e:
                logger.warning(f"Redis写入失败，降级到PostgreSQL: {e}")
                self._use_redis = False

        # 第二层：PostgreSQL（持久化存储）
        try:
            self.pg_store.save_conversation(session_id, messages, title)
        except Exception as e:
            logger.warning(f"PostgreSQL写入失败，降级到内存: {e}")
            self.memory.save(session_id, data)

    async def get_conversation(self, session_id: str) -> Optional[dict]:
        """获取会话（Redis → PostgreSQL → 内存）"""
        await self.connect()
        if self._use_redis:
            try:
                key = f"{self.KEY_PREFIX}{session_id}"
                data = await self.redis.get(key)
                if data:
                    return json.loads(data)
            except Exception as e:
                logger.warning(f"Redis读取失败，降级到PostgreSQL: {e}")

        # PostgreSQL 降级
        result = self.pg_store.get_conversation(session_id)
        if result:
            return result

        return self.memory.get(session_id)

    async def list_conversations(self, limit: int = 20, offset: int = 0) -> List[dict]:
        """获取会话列表（Redis → PostgreSQL → 内存）"""
        await self.connect()
        if self._use_redis:
            try:
                session_ids = await self.redis.zrevrange(self.LIST_KEY, offset, offset + limit - 1)
                return [await self.get_conversation(sid) for sid in session_ids if await self.get_conversation(sid)]
            except Exception as e:
                logger.warning(f"Redis列表读取失败，降级到PostgreSQL: {e}")

        result = self.pg_store.list_conversations(limit, offset)
        if result:
            return result

        sessions = self.memory.list_all()
        sessions.sort(key=lambda x: x.get("updated_at", ""), reverse=True)
        return sessions[offset:offset + limit]

    async def delete_conversation(self, session_id: str):
        """删除会话（三层存储均删除）"""
        await self.connect()
        self.memory.delete(session_id)
        if self._use_redis and self.redis:
            try:
                key = f"{self.KEY_PREFIX}{session_id}"
                await self.redis.delete(key)
                await self.redis.zrem(self.LIST_KEY, session_id)
            except Exception as e:
                logger.warning(f"Redis删除失败: {e}")
        try:
            self.pg_store.delete_conversation(session_id)
        except Exception as e:
```

# 10. 知识匹配度评估系统

### 10.1 什么是知识匹配度

知识匹配度（Confidence Score）是 LawBot+ 对每一条法律咨询回答可靠性的**量化评估**，取值范围为 **0.0 到 1.0**：

| 知识匹配度区间 | 含义 | 用户可感知表现 |
|-----------|------|--------------|
| **0.0** | 知识库中完全没有检索到相关法律条文，或非法律问题被检测到 | 直接返回"无法回答"，不触发人工审核 |
| **(0, 0.5)** | 检索结果相关性极低且分析质量差（经过 `content_penalty` 扣减后） | 风险等级为 high，进入人工审核（HITL） |
| **[0.5, 0.75)** | 有一定检索结果，但法律依据不够充分或数量不足 | 风险等级为 medium，正常返回 |
| **[0.75, 1.0]** | 检索结果高度可靠，分析严谨有据 | 风险等级为 low，正常返回 |

> **说明**：知识匹配度实际计算时受三项因子共同影响：基础分（avg 和 max 的加权）、文档数量奖励（+0.00~+0.10）和答案内容惩罚（-0.20），因此低分不一定由检索质量差单独决定，也可能由 AI 回答质量差（内容过短或含有"无法回答"等短语）触发。

知识匹配度的核心价值在于：**让用户一眼就知道这条回答有多可靠**。在法律场景中，这一点尤为重要——用户需要根据回答的可靠程度来决定是否采纳、是否需要进一步咨询律师。

### 10.2 知识匹配度的计算方式

LawBot+ 的知识匹配度采用**数学公式**计算，以检索指标为主，结合答案内容质量进行修正。

#### 评估维度

```
知识匹配度 = f(检索质量, 检索数量)
```

| 维度 | 说明 | 权重影响 |
|------|------|---------|
| **检索质量（avg）** | reranker 返回的平均相关度分数 | 主导因素（权重 0.6） |
| **检索质量（max）** | reranker 返回的最高相关度分数 | 次要因素（权重 0.4） |
| **文档数量** | 检索到的有效文档数量（1-5条奖励 +0.00~+0.10） | 辅助加分因子 |
| **答案内容质量** | `analysis_result` 是否含有"无法回答"等短语或长度不足50字（扣减 0.20） | 辅助惩罚因子 |

#### 核心公式（来自 `reviewer.py`）

```python
async def review(self, state: AgentState) -> AgentState:
    if not state.reranked_docs:
        # 检索结果为空时，知识匹配度=0，风险=low，不需要审核
        state.confidence_score = 0.0
        state.risk_level = "low"
        state.needs_review = False
        state.reasoning_chain.append("检索结果为空，知识匹配度=0.0")
        state.current_agent = "memory"
        return state

    rerank_scores = [doc.get("rerank_score", doc.get("score", 0)) for doc in state.reranked_docs]
    avg_score = sum(rerank_scores) / len(rerank_scores)
    max_score = max(rerank_scores)

    # 基础分 = 加权平均（avg 权重 0.6，max 权重 0.4）
    base = avg_score * 0.6 + max_score * 0.4

    # 文档数量奖励：1-5 条按比例奖励，最高 +0.10
    count_bonus = min(len(state.reranked_docs) / 5, 1.0) * 0.10

    # 答案内容惩罚：提及"无法回答"等短语或长度<50时降分 -0.20
    analysis = state.analysis_result or ""
    no_answer_phrases = [
        "无法回答", "无法提供", "无法找到", "暂无", "没有找到",
        "无法进行法律分析", "知识库中暂未找到",
    ]
    has_answer_content = not any(p in analysis for p in no_answer_phrases) and len(analysis) >= 50
    content_penalty = 0.0 if has_answer_content else -0.20

    raw_confidence = base + count_bonus + content_penalty
    state.confidence_score = max(0.0, min(1.0, round(raw_confidence, 3)))

    # 风险等级：>= 0.75 → low；0.50~0.75 → medium；< 0.50 → high
    if state.confidence_score >= 0.75:
        state.risk_level = "low"
    elif state.confidence_score >= 0.50:
        state.risk_level = "medium"
    else:
        state.risk_level = "high"

    # HITL 触发条件：知识匹配度低于阈值 OR 风险等级为 high/critical
    state.needs_review = (
        state.confidence_score < self.settings.hitl_confidence_threshold
        or state.risk_level in ("high", "critical")
    )
    return state
```

> **重要差异**（以代码为准）：
> 1. **知识匹配度公式**：相比文档描述的纯数学公式（`0.6 * max + 0.4 * avg`），代码实际增加了两项修正：
>    - `count_bonus`（文档数量奖励）：1-5 条按比例奖励，最高 +0.10，鼓励检索到更多相关法条
>    - `content_penalty`（答案内容惩罚）：当 `analysis_result` 含有"无法回答"等短语或长度不足 50 字时，强制扣减 0.20，防止 AI 敷衍回复获得高知识匹配度
> 2. **检索为空时行为**：文档描述风险等级为 `"high"` 且 `needs_review = True`，**代码实际**为 `risk_level = "low"` 且 `needs_review = False`，此时直接进入 `memory` 节点结束流程，不触发人工审核（因为无内容可审）。

**知识匹配度阈值**（`HITL_CONFIDENCE_THRESHOLD`）默认 **0.75**，低于此阈值的回答自动进入人工审核队列。

### 10.3 知识匹配度如何驱动流程分支

知识匹配度不是孤立显示的分数，它直接控制着工作流的走向：

```
                   ┌─────────────────────────┐
                   │      Reviewer Agent     │
                   │    输出: confidence_score │
                   │    输出: risk_level      │
                   │    输出: needs_review    │
                   └───────────┬─────────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
      confidence < 0.75   HIGH/CRITICAL    hitl_risk_keywords
      (配置阈值)           risk_level        命中任一
              │                │                │
              ▼                ▼                ▼
         needs_review      needs_review      needs_review
           = True            = True            = True
              │                │                │
              ▼                ▼                ▼
         HITL 节点        HITL 节点         HITL 节点
       (人工审核)        (人工审核)        (人工审核)
```

具体判断逻辑在 `reviewer.py` 中：

```python
# reviewer.py - 实际审核逻辑
state.needs_review = (
    state.confidence_score < self.settings.hitl_confidence_threshold
    or state.risk_level in ("high", "critical")
)
```

> **说明**：`settings.hitl_risk_keywords`（如"死刑""无期""判决"等）在 `settings.py` 中已定义，但 `reviewer.py` 当前版本的知识匹配度评估逻辑**未使用该字段**，仅基于 `confidence_score` 和 `risk_level` 触发审核。

当 `needs_review = True` 时，工作流进入 HITL（Human-In-The-Loop）节点，暂停等待人工审核员介入。

### 10.4 HITL 流程与触发条件

**触发条件**：满足以下任一条件即触发 `needs_review = True`：

1. `confidence_score < hitl_confidence_threshold`（默认 0.75）
2. `risk_level` 为 `high` 或 `critical`

**审核流程**：

```
用户提交问题 → LangGraph 工作流
     │
     ▼
Reviewer Agent 评估
     │
     ├─ confidence ≥ 0.75 且 risk_level == "low"
     │       ▼
     │   直接返回用户
     │
     └─ confidence < 0.75 或 risk_level == "high"/"critical"
             ▼
         needs_review = True
             │
             ▼
         HITL 人工审核队列（src/hitl/service.py）
             │
             ▼
         人工审核员审查答案 → 批准 / 修改 / 拒绝
             │
             ▼
         返回给用户
```

### 10.5 风险等级定义

| 等级 | 分值范围 | 触发条件 | 系统行为 |
|------|---------|---------|---------|
| **low** | ≥ 0.75 | 检索结果高度可靠 | 直接返回给用户 |
| **medium** | 0.5 ~ 0.75 | 有一定检索但不够充分 | 正常返回，可显示"建议核实"提示 |
| **high** | < 0.5 | 检索质量差或分析不可靠 | 自动触发 HITL 人工审核 |
| **critical** | (预留) | (类型定义存在，代码中未使用) | — |

#### 核心代码结构

```typescript
export function LegalAnswerCard({
  content,    // AI 返回的 Markdown 回答
  sources,    // 结构化来源列表
  confidence, // 知识匹配度 (0.0-1.0)
}: LegalAnswerCardProps) {
  // 1. 解析 Markdown 为结构化数据
  const parsed = useMemo(
    () => parseMarkdownToAnswer(content, sources),
    [content, sources]
  );

  // 2. 合并来源（避免法条与来源重复展示）
  const finalArticles = useMemo(() => {
    if (sources.length > 0) {
      return sources.map(s => ({
        title: s.title || s.article || "法律条文",
        content: s.content,
      }));
    }
    return parsed.legalArticles;
  }, [sources, parsed.legalArticles]);

  return (
    <div className="space-y-3">
      {/* 知识匹配度进度条 */}
      <ConfidenceBar confidence={confidence} />

      {/* A: 核心结论 */}
      <ConclusionCard content={parsed.conclusion} />

      {/* B: 核心要点 */}
      <KeyPointsCard points={parsed.keyPoints} />

      {/* C: 相关法条（可折叠） */}
      <LegalArticleAccordion articles={finalArticles} />

      {/* D: 参考来源 */}
      <SourcesCard sources={sources} />
    </div>
  );
}
```

### 11.5 Markdown 解析器

**函数**: `parseMarkdownToAnswer(markdown, sources)`

将 AI 返回的 Markdown 文本按 `## 二级标题` 分块，智能提取：

| 区块类型 | 识别关键词 | 解析逻辑 |
|---------|-----------|---------|
| **结论** | "结论""总结""一句话" | 取该区块下所有文本 |
| **要点** | "要点""建议""核心" | 按 `1.` / `**粗体标题**` 切分为列表 |
| **法条** | "法条""法律依据" | 按双换行分段，每段首行为标题 |

解析优先级：`sources` 字段（结构化数据）> Markdown 中提取的法条。

#### 粗体标记清理

AI 返回的原始 Markdown 包含 `**粗体**` 语法，这些标记在解析和渲染两个阶段被清理，避免用户看到裸标记符号：

| 清理位置 | 代码 | 说明 |
|---------|------|------|
| `boldMatch` 分支（第 105 行） | `boldMatch[3].trim().replace(/\*\*/g, "")` | 要点 body 去掉 `**` |
| `numMatch` 分支（第 110 行） | `numMatch[2].trim().replace(/\*\*/g, "")` | 无粗体格式的行兜底 |
| `ConclusionCard`（第 157 行） | `content.replace(/\*\*/g, "")` | 结论全文清理 `**` |
| `KeyPointsCard` 渲染层（第 199 行） | `point.body.replace(/\*\*/g, "")` | 输出前兜底保险 |

### 11.6 样式细节

#### 粗体样式（`strong` 标签）

核心要点的粗体文字在 `KeyPointsCard` 中渲染，使用 `text-base`（16px）比父元素 `text-sm`（14px）大一档，配合 `font-bold` 加粗：

#### 问题：后端(8000)和前端(3000)端口被占用

**问题描述**

启动服务时提示端口被占用，后端无法在 8000 端口启动，前端无法在 3000 端口启动。

**排查过程**

1. 使用 `netstat -ano | findstr ":8000"` 检查 8000 端口，发现有进程(PID 30352)在监听
2. 使用 `netstat -ano | findstr ":3000"` 检查 3000 端口，发现有进程(PID 21724)在监听
3. 确认这些是之前启动未正常关闭的服务进程

**解决方案**

```powershell
# 1. 通过 PID 强制终止占用端口的进程
taskkill /PID 30352 /F   # 终止后端 8000 端口进程
taskkill /PID 21724 /F   # 终止前端 3000 端口进程

# 2. 确认端口已释放
netstat -ano | findstr ":8000"
netstat -ano | findstr ":3000"

# 3. 重新启动服务
# 后端 (src 目录)
python -m src.main api

# 前端 (frontend 目录)
npm run dev
```

**启动后验证**

- 后端 API: `http://localhost:8000` — Uvicorn 服务正常，BM25索引和Embedding模型加载完成，343篇法律文档知识库就绪
- 前端: `http://localhost:3000` — Next.js 开发服务器正常

**预防措施**

- 使用 `run.bat` 脚本统一管理服务生命周期，避免手动启动导致端口残留
- 或在启动新服务前，先检查并清理旧进程

#### 问题：AI 回答中显示原始 `**` 粗体标记符号

**问题描述**

用户看到的 AI 回答中大量显示 Markdown 粗体标记符号 `**`，如 `**工伤认定三要素**`，严重影响阅读体验。这些 `**` 应该被解析为加粗效果，而不应该直接显示出来。

**根本原因**

`parseMarkdownToAnswer` 解析器在提取要点列表时，正则 `boldMatch` 将 `**` 包围的内容捕获为 `title`，但 `body` 部分（`.*`）保留了剩余文本中的 `**` 标记，导致渲染时原始语法泄露。此外 `ConclusionCard` 直接渲染未经清理的 `content` 文本。

**解决方案**

在解析和渲染两个阶段全局清理 `**` 标记：

1. `boldMatch` 分支 — `body` 字段加 `.replace(/\*\*/g, "")`（第 105 行）
2. `numMatch` 分支兜底 — 同上（第 110 行）
3. `ConclusionCard` — 渲染前用 `content.replace(/\*\*/g, "")` 清理（第 157 行）
4. `KeyPointsCard` 渲染层 — `point.body` 输出前再清一遍兜底（第 199 行）

**涉及文件**

- `frontend/components/chat/legal-answer-card.tsx`

**修改后效果**

```
修改前：1. **工伤认定三要素** 工作时间、工作场所...
修改后：1. 工伤认定三要素  工作时间、工作场所...
```

粗体标题通过 `strong` 标签的 `font-bold text-base` 类正常加粗显示，不再有原始标记。

### 12.10 RAGAS 离线批量评估管线

**日期**：2026-04-28

**功能描述**：实现"无侵入埋点 + 离线批量评估"的 RAGAS 管线，主业务 P99 延迟不受影响，定期输出忠实度、精确度、召回率指标报告。

#### 架构总览

```
用户提问
    │
    ▼
+------------------------------------------------------------+
|  LangGraph 工作流（run_legal_consultation）               |
|  ... -> relevance_check -> analyst -> reviewer ->           |
|  memory -> finalize_node（末尾）                          |
|                       |                                   |
|                       V  log_rag_eval_data.delay()       |
|           +-----------------------------+                |
|           |  RetrievalEvalLog | PostgreSQL                |
|           +-----------------------------+                |
+------------------------------------------------------------+
    |
    | 定期离线执行
    V
python eval/run_ragas_eval.py
    |
    +-- 读取最近 N 天埋点数据
    +-- 转换为 RAGAS Dataset
    +-- Judge LLM（qwen-plus）评估
    +-- -> 终端报告 + data/ragas_report_YYYYMMDD.csv
```

#### 1. 异步埋点任务

**文件**：`src/tasks/rag_eval.py`

Celery 任务 `log_rag_eval_data(query, reranked_docs, final_answer, retrieved_docs)`：

| 写入字段 | 来源 |
|---------|------|
| `query` | 用户原始问题 `state.user_input` |
| `reranked_doc_ids` | `reranked_docs` 中提取 `title::article` 组合 |
| `retrieved_doc_ids` | `retrieved_docs` 中提取 `title::article` 组合（重排前） |
| `relevance_scores` | `reranked_docs` 中每条文档的 `rerank_score/score` |
| `final_selected_doc_id` | `reranked_docs[0].id`（top-1 文档） |
| `final_answer` | AI 最终回答 |
| `user_feedback` | 默认为 `None`（后续由人工标注补充，分数范围 1-5 分） |

任务装饰器：
```python
@celery_app.task(
    bind=True,
    name="lawbot.log_rag_eval_data",
    max_retries=3,
    default_retry_delay=10,
    autoretry_for=(Exception,),
    retry_backoff=True,
)
```

- 异常仅打 `logger.warning`，不向上传播
- Celery 自动重试（最多 3 次），超过后任务标记为 FAILURE
- 主业务流完全不受影响

#### 2. 业务流接入

**文件**：`src/agents/workflow.py` — `finalize_node` 末尾

```python
# ===== RAG 评估埋点（无侵入，不阻塞主流程）=====
try:
    from src.tasks.rag_eval import log_rag_eval_data
    log_rag_eval_data.delay(
        query=state.user_input,
        reranked_docs=state.reranked_docs,
        final_answer=state.final_answer,
        retrieved_docs=state.retrieved_docs,
    )
except Exception as hitl_exc:
    logger.warning(f"[RAGEval] 埋点失败，不影响主流程: {hitl_exc}")
```

`finalize_node` 是 `async def`，`log_rag_eval_data.delay()` 是同步调用、立即返回消息 ID，**不等待执行结果**，绝对不阻塞响应。

#### 3. RAGAS 离线评估脚本

**文件**：`eval/run_ragas_eval.py`

用法：
```bash
python eval/run_ragas_eval.py                    # 默认最近 30 天
python eval/run_ragas_eval.py --days 7           # 最近 7 天
python eval/run_ragas_eval.py --output my.csv   # 指定导出路径
python eval/run_ragas_eval.py --model qwen-max  # 指定 Judge LLM
```

**依赖检查**：脚本开头检测 `ragas` 是否安装，未安装时打印提示并以 exit code 1 退出。

**数据转换**（适配 RAGAS `Dataset` 格式）：

| RAGAS 字段 | 来源 |
|-----------|------|
| `user_input` | `RetrievalEvalLog.query` |
| `response` | `RetrievalEvalLog.final_answer` |
| `retrieved_contexts` | `reranked_doc_ids` -> `List[List[str]]`，每个元素为 `"title::article"` |
| `ground_truth` | `user_feedback` 非空则用评分；否则占位 `["暂无人工标注"]` |

**Judge LLM 初始化**：
```python
judge_llm = ChatOpenAI(
    model="qwen-plus",
    temperature=0.1,
    base_url=settings.dashscope_base_url,
    api_key=settings.dashscope_api_key,

```

| 指标 | 说明 |

| `ContextPrecision` | 上下文的精确度（top-k 块中相关块的比例和排序质量） |
| `ContextRecall` | 上下文的召回率（上下文覆盖了多少标准答案中的信息） |
